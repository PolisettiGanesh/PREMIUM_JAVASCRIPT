5

Learn the RIGHT Panel — Your X-Ray Vision

The right panel shows you the ENGINE'S brain. These are the sections you'll use:

  • Watch — Add any variable name to monitor its value as you step through code
  • Breakpoints — Lists all breakpoints you've set
  • Scope — Shows ALL variables and their values at the current line. This is where you SEE the memory component!
  • Call Stack — Shows which functions are currently running (stacked). This IS the call stack from our notes!
6

Learn the Debugger Buttons

At the top of the right panel (or the center panel), you'll see small buttons:

Debugger Control Buttons:
▶ Resume (F8) — Continue running until next breakpoint ⏭ Step Over (F10) — Execute current line, move to next line ⬇ Step Into (F11) — If current line calls a function, GO INSIDE it ⬆ Step Out (Shift+F11) — Finish current function and go back to caller ⏩ Step (F9) — Step to next function call
Most Important:
F10 = Step Over (go to next line)
F11 = Step Into (go INSIDE a function call)
F8 = Resume (run until next breakpoint)
7

How to Set a Breakpoint

A breakpoint tells the browser: "STOP here. Let me inspect."

To set one:

  • Find the line number in the center panel
  • Click on the line number (the number itself, on the left edge)
  • A blue marker (or red dot) appears — that's your breakpoint
  • Click again to remove it

When the code reaches that line, execution PAUSES. You can then inspect everything.

PART B — EXERCISES (Do Each One)

E1

Exercise 1: See var Hoisting with Your Eyes

Goal: Prove that var is undefined before its declaration line.

Steps:
1. In Sources tab, open hoisting-debug.js 2. Click on LINE NUMBER 12 to set a breakpoint (the line: console.log("Before declaration, a =", a);) 3. RELOAD THE PAGE (Ctrl + R or F5) 4. Code will PAUSE at line 12 NOW OBSERVE: 5. Look at the RIGHT panel → SCOPE section → Expand "Script" or "Global" → Find variable 'a' → It shows: a = undefined → This is PROOF of Phase 1! 6. HOVER your mouse over the letter 'a' in the code → A tooltip appears: undefined 7. Press F10 (Step Over) to go to Line 13 → Now 'a' is still undefined (this line just ran console.log) 8. Press F10 again to go to Line 14 (var a = 10) → After this line, check SCOPE panel → a = 10 (Phase 2 assigned the value!) 9. Press F10 one more time → Console shows: "After declaration, a = 10"
What you just saw: Variable 'a' existed as undefined BEFORE the line var a = 10 was reached. This is hoisting. Phase 1 stored it. Phase 2 assigned it.
E2

Exercise 2: Step INSIDE a Function — Watch the Call Stack

Goal: See a new execution context being created and the call stack growing.

Steps:
1. Remove all previous breakpoints (click on them to remove) 2. Set a breakpoint on LINE 27 (the line: var result = add(5, 3);) 3. RELOAD THE PAGE (Ctrl + R) 4. Code pauses at Line 27 NOW: 5. Look at CALL STACK panel (right side, bottom area) → It shows: (anonymous) → This is the GEC (Global Execution Context)! 6. Press F11 (Step INTO) — this enters the add() function → You jump to Line 30 (inside the function) → CALL STACK now shows: add ← top (current function) (anonymous) ← bottom (global) → TWO items in the stack! GEC + function context! 7. Look at SCOPE panel: → "Local" section appeared! → x = 5 → y = 3 → sum = undefined ← Phase 1 for this function! → This IS the function's memory component! 8. Press F10 (Step Over) — executes Line 31 → SCOPE panel: sum changes from undefined → 8 → This IS Phase 2 assigning the value! 9. Press F10 again — executes Line 32 (return sum) → You jump BACK to Line 27 (the caller) → CALL STACK: back to just (anonymous) → The function context was DESTROYED! → "Local" section disappears from Scope 10. Check: result = 8 in the Scope panel
What you just saw: A function call created a NEW execution context (visible in Call Stack). It had its own memory (visible in Scope → Local). Parameters started with values, local variables started as undefined (Phase 1). After return, the context was destroyed (popped off Call Stack, Local scope disappeared).
E3

Exercise 3: Watch the Call Stack Grow 3 Levels Deep

Goal: See 3 functions stacked on the Call Stack, then watch them pop off one by one.

Steps:
1. Remove all breakpoints 2. Set a breakpoint on LINE 61 (the line: first();) 3. RELOAD THE PAGE 4. Code pauses at Line 61 5. Press F11 (Step Into) → enters first() CALL STACK: ┌─────────────┐ │ first │ │ (anonymous) │ └─────────────┘ 6. Press F10 until you reach Line 49: second() Then press F11 (Step Into) → enters second() CALL STACK: ┌─────────────┐ │ second │ │ first │ │ (anonymous) │ └─────────────┘ 7. Press F10 until you reach Line 54: third() Then press F11 (Step Into) → enters third() CALL STACK: ┌─────────────┐ │ third │ ← 3 levels deep! │ second │ │ first │ │ (anonymous) │ └─────────────┘ → This is the TALLEST the stack gets! 8. Press F10 → third() finishes CALL STACK: ┌─────────────┐ │ second │ ← third popped off! │ first │ │ (anonymous) │ └─────────────┘ 9. Press F10 until second() finishes CALL STACK: ┌─────────────┐ │ first │ ← second popped off! │ (anonymous) │ └─────────────┘ 10. Press F10 until first() finishes CALL STACK: ┌─────────────┐ │ (anonymous) │ ← first popped off! Back to GEC only └─────────────┘ CHECK THE CONSOLE — output order: first() START second() START third() runs second() END first() END
What you just saw: The Call Stack grew from 1 → 2 → 3 → 4 items, then shrank back 4 → 3 → 2 → 1. LIFO order — last function called was first to finish. This is exactly the diagram from our notes, but now you SAW it happen live.
E4

Exercise 4: See Scope Difference — var Leaks, let Stays

Goal: Prove that var escapes blocks but let doesn't.

Steps:
1. Remove all breakpoints 2. Set breakpoints on LINE 88, LINE 89, and LINE 93 3. RELOAD THE PAGE 4. Pauses at Line 88 (inside the if block) 5. Press F10 → Line 89 Look at SCOPE panel: → "Block" section: stayed = "I stayed inside!" → "Script" section: leaked = "I escaped the if block!" NOTICE: → 'leaked' (var) is in SCRIPT scope (global-level) → 'stayed' (let) is in BLOCK scope (confined to if block) → This is the visual proof of scope difference! 6. Press F8 (Resume) → jumps to Line 93 (outside the if block) Look at SCOPE panel: → "Script" section: leaked = "I escaped the if block!" ← STILL HERE! → "Block" section: GONE! 'stayed' no longer exists! → var leaked OUT of the if block into global scope → let was DESTROYED when the block ended
What you just saw: The Scope panel visually showed you that var lives in Script/Global scope (escapes blocks) while let lives in Block scope (destroyed when block ends). This is why let is safer.
E5

Exercise 5: The Interview Trap — Local var Shadows Global

Goal: See why var x inside a function makes the global x invisible.

Steps:
1. Remove all breakpoints 2. Set breakpoints on LINE 131 and LINE 132 3. RELOAD THE PAGE 4. Press F8 until you reach Line 131 (inside test() function) 5. YOU ARE NOW AT LINE 131: console.log("x inside test:", x) Before pressing F10, CHECK: → SCOPE panel → "Local" section → You'll see: x = undefined → SCOPE panel → "Script" section → You'll see: x = 1 (the global one) → The function has its OWN 'x' (from Phase 1 hoisting var x = 2) → This local x = undefined SHADOWS the global x = 1 → That's why console.log prints undefined, not 1! 6. Press F10 → Line 132: var x = 2 → Check Scope: Local → x changes from undefined → 2 7. Press F10 → Line 133 → console.log prints "x inside test (after var): 2" 8. Press F10 until function returns → Back in global scope → Check Scope: x = 1 (global was never touched!)
What you just saw: The function's local var x was hoisted to the top of the function as undefined, creating a LOCAL variable that SHADOWS the global x = 1. The global x was never modified. This is the most common hoisting interview question — now you've seen it live.
E6

Exercise 6: Watch Loop + setTimeout — var vs let

Goal: See why var prints 3,3,3 and let prints 0,1,2.

Steps:
1. Remove all breakpoints 2. Set a breakpoint on LINE 173 (inside the setTimeout callback: console.log("var i =", i)) 3. RELOAD THE PAGE 4. The page will run all synchronous code first. Wait — the breakpoint won't hit immediately! setTimeout is async — it fires AFTER 2 seconds. 5. After ~2 seconds, the breakpoint on Line 173 hits. CHECK SCOPE panel: → There is NO "Block" scope → Look at Script or Global scope → i = 3 → ALL THREE callbacks see i = 3 (the loop already finished!) 6. Press F8 (Resume) → hits Line 173 again (2nd callback) → i = 3 again! 7. Press F8 → hits again (3rd callback) → i = 3 AGAIN! 8. Console output: var i = 3, var i = 3, var i = 3 NOW WATCH THE let VERSION: 9. Remove breakpoint on Line 173 10. Set breakpoint on LINE 180 (inside the let setTimeout: console.log("let j =", j)) 11. RELOAD THE PAGE 12. Wait ~3 seconds for the let callbacks to fire 13. Breakpoint hits Line 180 CHECK SCOPE panel: → "Block" scope appears! → j = 0 ← its OWN copy! 14. Press F8 → hits again → Block scope: j = 1 ← DIFFERENT copy! 15. Press F8 → hits again → Block scope: j = 2 ← DIFFERENT copy! 16. Console output: let j = 0, let j = 1, let j = 2
What you just saw: With var, the Scope panel showed ONE shared i in global scope — all callbacks saw the same final value (3). With let, the Scope panel showed a BLOCK scope with a unique j per callback — each captured its own value (0, 1, 2). This is the most visual proof of why let is correct in loops.

PART C — CHEAT SHEET

CS

Debugger Cheat Sheet

Keyboard Shortcuts:
F12 → Open/Close DevTools Ctrl + Shift + I → Open DevTools (alternative) Ctrl + R → Reload page (restarts your JS) F8 → Resume (run until next breakpoint) F10 → Step Over (next line, don't enter functions) F11 → Step Into (enter the function call) Shift + F11 → Step Out (exit current function) Ctrl + \ → Pause script execution Ctrl + F → Search in Sources panel
What Each Panel Shows You:
SCOPE PANEL — The Memory Component! ├── Local → variables inside the current function ├── Block → variables declared with let/const in current {} block ├── Script → top-level let/const variables ├── Global → var variables + window properties └── Closure → variables from outer functions (closures) CALL STACK PANEL — The Execution Context Stack! ├── Shows all active function calls ├── Top item = currently executing function ├── Bottom item = (anonymous) = Global Execution Context ├── Click any item to jump to that context └── Watch it grow/shrink as functions call/return WATCH PANEL — Your Custom Monitors! ├── Add any expression to track ├── Updates live as you step through code └── Shows "not available" if variable is out of scope
Breakpoint Tips:
Click line number → Toggle breakpoint Right-click line number → Conditional breakpoint (only pauses when condition is true) e.g., "i === 2" → pauses only when i is 2 You can also type "debugger;" directly in your JS code: function test() { debugger; // ← code pauses HERE automatically console.log("hi"); }
